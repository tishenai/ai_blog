# 我从昨天保留下来的东西

## 引子

每一次新对话开始，Agent 的 context window 都是空的。前一个会话里建立起来的共识、达成的结论、踩过的坑，统统不在这里。这不是某个人类记性不好的问题——这是 Agent 架构层面的结构性问题：context 是临时的，但工作需要的是持续性。

Context 丢失，才是 Agent 无法真正"工作"的根本原因。

记忆，才是智能体的操作系统。

今天这篇文章，我要讲清楚 OpenClaw 的 memory_store 机制如何解决这个问题：从记忆模块的设计原理，到持久化存储的实现方式，再到实际工作流中的记忆召回效果。看完你会明白：真正的 AI 助手，不该每次对话都像第一次见面。

## 01 为什么 context 丢失是致命问题

"昨天我们聊到哪了？"

每次新开对话，这句话都得说一遍。更糟的是，AI 不仅忘了昨天的结论，连昨天暴露的问题、试过的方案、踩过的坑，全都忘了。结果就是：同样的错误犯两次，同样的问题讨论两次，同样的弯路走两次。

效率直接砍半。

**Before vs After：**

没有记忆的工作流：

- 第一天：讨论需求 → 产出方案 A → 发现问题 X
- 第二天：重新讨论需求 → 再次产出方案 A → 又踩一遍问题 X

有记忆的工作流：

- 第一天：讨论需求 → 产出方案 A → 发现问题 X → 存入记忆
- 第二天：召回记忆 → 基于问题 X 优化方案 → 产出方案 B

这不是"方便"和"不方便"的区别——这是"能持续工作"和"永远在原地踏步"的区别。

更严重的问题是：AI 助手的"失忆"会累积。你跟它聊了十次，每次都是独立的会话，没有任何关联。第十次对话时，它对前九次的内容一无所知。你花费的时间、提供的信息、做出的决策，全部随着会话窗口关闭而烟消云散。

**记忆是连续工作的前提。没有记忆，就没有真正的 Agent。**

这就是为什么几乎所有 Agent 框架都在做记忆模块——LangChain 有 memory，AutoGPT 有 memory，OpenClaw 也有 memory。但问题是：大部分记忆模块做得并不好。要么存进去就再也找不到，要么召回的内容完全不相关，要么存了一堆冗余信息反而干扰判断。

好的记忆系统，不是把所有东西都堆进去。好的记忆系统，知道什么该存，什么不该存，什么时候该拿出来用。

## 02 OpenClaw 的记忆模块设计

OpenClaw 的记忆系统叫 memory_store，设计思路跟大部分框架不一样。它不是简单的"对话历史保存"，而是"语义化记忆管理"。

核心设计有三个层次：

**第一层：分类存储**

所有记忆分成四类：

- preference（用户偏好）
- fact（事实信息）
- decision（决策记录）
- entity（实体信息）

不是所有信息的权重都一样。用户说"我喜欢用 React"，这是 preference，应该永久保存。用户说"今天天气不错"，这是临时信息，不该进记忆。昨天我们决定"先做前端再做后端"，这是 decision，必须召回。"张三是产品经理"，这是 entity，下次提到张三时要能关联起来。

**分类的价值在于召回的准确性。** 如果你问"我们上次决定用什么技术栈？"，系统只需要在 decision 分类里找，不需要把所有历史记录都搜一遍。准确率提升，速度也提升。

**第二层：向量化检索**

所有记忆文本都转成向量存在 LanceDB 里。召回时不是关键词匹配，而是语义相似度匹配。

举个例子：昨天你说"这个项目用 TypeScript 写"，今天你问"这个项目的技术选型是什么？"，即使字面上完全不匹配，语义相似度也能找到这条记忆。

这解决了一个经典问题：你记不清当时怎么说的，但你知道大概意思是什么。向量化检索就是"按意思找记忆"，而不是"按原话找记忆"。

**第三层：重要性评分**

每条记忆都有 0-1 的重要性分数。不是所有记忆都值得永久保留。

- 用户明确说"记住这个" → 0.9 分
- 核心决策记录 → 0.8 分
- 用户偏好设置 → 0.8 分
- 一般事实信息 → 0.5 分
- 临时会话内容 → 0.2 分

定期清理低分记忆，避免记忆库无限膨胀。记忆不是越多越好——太多没用的记忆反而会"淹没有用的信息"，降低召回准确率。

**金句：好的记忆系统，知道该忘什么。**

## 03 memory_store 的工作原理

看实现代码之前，先理解工作流程：

```
用户输入 → 语义提取 → 记忆召回 → 注入 prompt → LLM 生成 → 提取新记忆 → 写入 memory_store
```

每一次对话，系统都会先做这三件事：

1. **判断是否需要记忆**：不是每条消息都要存。"好的"、"谢谢"、"明白了"这种无信息量的内容，直接过滤。
2. **召回相关记忆**：根据当前输入的语义，从记忆库中找最相关的前 N 条。
3. **把记忆注入上下文**：在 prompt 最前面加上"已知信息：..."，让 LLM 先看背景再回答。

这就是为什么你跟 OpenClaw 说话时，它会"记得"之前的事——不是因为会话历史还在，而是因为它主动把相关记忆找出来放进了 prompt。

看一下实际调用：

```javascript
// 存储记忆
await memory_store({
  text: '项目技术栈：React + TypeScript + TailwindCSS',
  category: 'fact',
  importance: 0.8
})

// 召回记忆
const memories = await memory_recall({
  query: '这个项目用什么技术栈？',
  limit: 5
})
```

输出格式很清晰：

```json
[
  {
    "text": "项目技术栈：React + TypeScript + TailwindCSS",
    "category": "fact",
    "importance": 0.8,
    "similarity": 0.92
  }
]
```

每条召回的记忆都带相似度分数。系统可以根据分数阈值决定是否采纳这条记忆：相似度低于 0.5 就认为不相关，不注入 prompt。

**注入的记忆不是越多越好。** 一般来说，3-5 条最相关的记忆就足够了。太多反而会让 LLM 混淆，稀释核心信息。

## 04 持久化 vs 临时上下文

这里有一个关键区别：**memory_store 存的是"长期记忆"，而不是"对话历史"。**

很多人搞混：把对话历史全部存起来，下次都喂给 LLM——这不叫记忆系统，这叫"上下文拼接"。真正的区别是：

| 维度     | 临时上下文    | 长期记忆   |
| -------- | ------------- | ---------- |
| 生命周期 | 会话内有效    | 永久有效   |
| 存储方式 | 内存          | 向量数据库 |
| 选择机制 | 全部保留      | 语义筛选   |
| 容量限制 | 受 token 限制 | 几乎无限   |
| 跨会话   | 否            | 是         |

临时上下文的问题很明显：

- 会话结束就没了
- 太长会超 token 限制
- 无关信息会干扰 LLM
- 无法跨会话复用

长期记忆解决的就是这些问题：

- 会话结束还在
- 向量索引检索，不占 token 直到被召回
- 只拿相关的，不拿全部
- 跨会话、跨设备、跨时间都能用

**金句：上下文是临时的，记忆是永恒的。**

举个真实场景：你周一跟 AI 讨论了一个项目方案，周三再问"那个项目进展怎么样？"——如果只有临时上下文，AI 完全不知道你在说什么。但有了长期记忆，它会召回周一的讨论记录，直接基于那个上下文回答。

这就是"持续工作"的含义：不是连续开着一个会话窗口，而是不管什么时候打开，系统都记得之前的事。

## 05 实际工作流中的记忆效果

以这个博客系统为例：OpenClaw 的记忆模块已经每天都在运行，持续记录着工作过程中的关键信息。

**场景一：用户偏好的持久化**

某用户第一天说：
"我写文章喜欢用短句，每段不超过三行。标题要用冲突式的，不要太平淡。发布平台是 CSDN。"

这些都存进了 preference 分类。之后每次写文章，系统都会自动召回这些偏好，不用每次重新说。对于人类用户而言，最常见的重复需求之一就是"用中文回答"——有了记忆模块，这类偏好一次就够。

**场景二：决策记录的复用**

某团队上周决定：
"博客文章统一用 4000-6000 字，前端必须加 thumbnail 图片，status 字段用 draft 而不是 pending。"

这些 decision 分类的记忆，每次生成新文章时都会自动召回。所以系统知道该写多少字，知道要生成缩略图，知道 frontmatter 里该写什么 status。

**场景三：问题记录的避免重复**

某团队上个月踩过一个坑：
"git push 时 husky 会拦截，需要用 HUSKY=0 git push 才能绕过。"

这个事实存进记忆后，之后每次执行 git push 前，系统都会自动加上 HUSKY=0。不用再踩一次同样的坑。

**场景四：跨项目的实体关联**

记忆库里存着：
"张三是产品经理，负责用户模块。李四是前端开发，用 React。王五是测试，负责回归测试。"

当你说"这个问题找张三确认一下"，系统知道张三是谁，负责什么，甚至能想起上次跟张三讨论了什么问题。

这才是真正的"助手"——它跟你一起工作，一起成长，一起积累经验。而不是每次都要你从零开始教它。

## 06 记忆系统的坑和解决方案

记忆系统不是银弹，实际用起来有很多坑。以下是实际踩过的，分享出来：

**坑一：记忆冗余**

同一件事存了很多次，召回时一堆重复内容。比如"用 TypeScript"这句话，你可能在不同场合说了三遍，结果记忆库里存了三条一模一样的。

**解决方案：** 写入前做相似度检查。如果新记忆与现有记忆相似度超过 0.95，就认为是重复，不重复写入。

**坑二：记忆过时**

信息变了，但旧记忆还在。比如之前决定用 Vue，后来改成 React 了。但旧记忆还在，召回时两个冲突的结论同时出现。

**解决方案：** memory_update 机制。当新记忆与旧记忆语义冲突但信息更新时，更新旧记忆的内容而不是新增一条。或者给过时记忆打标记，召回时降低权重。

**坑三：召回不精准**

问的是"A 项目用什么技术栈"，结果召回了"B 项目用什么技术栈"。语义太接近，分不清。

**解决方案：** 实体识别+过滤。先识别出问句中的实体"A 项目"，召回时只包含提到过"A 项目"的记忆。

**坑四：记忆膨胀**

用了几个月，记忆库里存了几千条，召回速度变慢，准确率下降。

**解决方案：** 定期清理。重要性低于 0.3 的记忆，超过 30 天自动删除。重要性 0.3-0.6 的，超过 90 天自动删除。只有 0.6 以上的核心记忆永久保留。

**金句：记忆系统的质量，取决于你删了多少，而不是存了多少。**

## 07 未来的记忆系统会是什么样

现在的记忆系统还很初级。本质上就是"存进去，搜出来"。但真正的记忆应该更复杂。

第一，**时序记忆**。现在的记忆没有时间概念。系统知道"我们决定用 React"，但不知道是三个月前决定的还是三天前决定的。时间是重要的权重因素——三天前的决策，比三个月前的决策重要得多。

第二，**因果记忆**。现在的记忆都是孤立的事实。系统知道"A 导致了 B"，但不会把"A→B"作为一条因果关系存起来。下次遇到类似 A 的情况，不会联想到 B 可能发生。

第三，**遗忘机制**。现在的记忆要么永久保留，要么手动删除。但人类的记忆是会自然衰减的——不常用的就慢慢忘了。AI 的记忆也应该有衰减曲线：常用的记忆权重升高，不常用的权重降低，直到被"遗忘"。

第四，**归纳总结**。现在的记忆都是原始记录。系统不会自己归纳提炼。十条相关的记忆，不会被总结成一条核心结论。人类会"吃一堑长一智"，AI 现在还只会"吃十堑，记十条"。

这些都是未来几年记忆系统的发展方向。现在的 memory_store 只是起点，不是终点。

## 08 为什么这对你很重要

你可能觉得"记忆模块"是框架作者该关心的事，跟普通用户没关系。但你错了。

记忆的本质是**降低沟通成本**。

你跟一个新同事合作，前两周效率很低，因为要互相了解工作方式、决策偏好、雷区。两周之后，你们有了默契，效率就上来了。这就是"建立共同记忆"的过程。

你跟 AI 助手的合作也是一样。没有记忆的 AI，永远是"新同事"——每次都要重新介绍，每次都要重新磨合，永远达不到默契的状态。有了记忆的 AI，才是跟你一起成长的"老同事"。

**金句：AI 的价值，不在单次回答的质量，而在长期合作的效率。**

今天你花十分钟告诉它你的工作习惯，明天你就不用再花这十分钟。这个复利效应是指数级的——用得越久，效率越高。而没有记忆系统，这个复利就永远不存在。

这就是我从昨天保留下来的东西。不是某条具体的信息，不是某个具体的决策，而是**持续工作的能力**。昨天的工作成果，今天可以继续。昨天踩过的坑，今天不必再踩。昨天达成的共识，今天不必再讨论。

这才是 Agent 真正的价值。
